# Roundcube Webmail Sidecar

Roundcube is the real browser mail client for the reference Mailcow email
module. It replaces the earlier custom demo-only webmail page while preserving
the same platform evidence path:

`Roundcube -> Report Phish plugin -> /demo-report -> Mailcow quarantine -> dashboard intake -> iTop sync -> agent`

The hidden `/demo-report` endpoint must both create the dashboard/iTop ticket
and ensure an investigation agent is started. It first lets the dashboard RACI
auto-assignment path run; if that does not return `assigned`, it uses the
runtime `DASHBOARD_SERVICE_TOKEN` to call `/api/tickets/{id}/assign-agent`.
This prevents demo reports from stopping at "ticket created" without an actual
agent investigation.

## Reference Deployment

Live AI server defaults:

- Public demo URL: `http://127.0.0.1:2581/webmail/`
- Compatibility route: `http://127.0.0.1:2581/SOGo/so` redirects to `/webmail/`
- Roundcube sidecar: `roundcube-mailcow-demo`
- Loopback sidecar port: `127.0.0.1:2582`
- API/UI sidecar: `nginx-mailcow-api`
- PHP endpoint sidecar: `php-fpm-mailcow-api`
- Hidden report endpoint: `/demo-report`
- Report token file: `/opt/agentic-it/Mailcow/deploy/api-nginx/.roundcube_report_token`

The report token is generated by the deployer and must never be printed or
committed.

## Deploy Or Repair

Use server-manager for remote execution:

```bash
python "C:/Users/me/.agents/skills/server-manager/ssh_client.py" --server ai --script '
cd /opt/agentic-it/Mailcow/deploy
python3 scripts/deploy_mailcow_api.py
'
```

Expected deployer checks:

- Mailcow admin UI on `:2581` renders.
- Cache assets load.
- Stale Mailcow user sessions recover to the admin login.
- Mailcow/DataTables JSON endpoints return valid JSON.
- `/webmail/` renders Roundcube.
- `/SOGo/so` lands on Roundcube through the compatibility route.

The deployer only recycles sidecars it owns:

- `php-fpm-mailcow-api`
- `nginx-mailcow-api`
- `roundcube-mailcow-demo`

It must not restart unrelated agent processes or unrelated services.

## Troubleshooting

- If `/webmail/` returns `502`, check `docker logs roundcube-mailcow-demo` and
  verify port `127.0.0.1:2582` is listening.
- If the Report Phish button is missing, verify
  `ROUNDCUBEMAIL_PLUGINS=archive,zipdownload,report_phish` and that the plugin
  directory is mounted at `/var/www/html/plugins/report_phish`.
- If the button label shows `[reportphish]`, redeploy the plugin; the JS label
  patch should render `Report Phish`.
- If reporting returns an error, validate `/web/mailcow_demo_report.php`, the
  token file, and `php-fpm-mailcow-api` logs.
- If the dashboard ticket is created but the agent blocks on Wazuh access, that
  is expected least-privilege behavior; approve or route the access request
  instead of bypassing the gate.

## Security Notes

- Do not commit the Roundcube report token, mailbox passwords, API keys, or
  Mailcow database credentials.
- The token is file-backed under `api-nginx/.roundcube_report_token` with mode
  `600`.
- SQL writes use parameterized PDO in the sidecar endpoint.
- Dashboard/iTop sync and agent actions remain auditable through the platform.
